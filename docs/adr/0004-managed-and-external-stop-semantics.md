# ADR-0004：托管进程与外部进程的停止语义

- 状态：Accepted
- 日期：2026-07-14
- 决策者：Dev Process Manager 项目
- 关联任务：P0-T01、P0-T02、P0-T03、P4、P6-T05
- 替代：无
- 被替代：无

## 背景

工具启动的托管进程与扫描发现的外部进程具有不同的控制证据。托管进程可在创建时建立 Windows Job Object 或 macOS 进程组；外部进程通常只有瞬时观察结果，递归遍历 PID 无法可靠证明整棵树仍属于同一任务。把两者放在同一停止接口会夸大控制能力并增加误杀风险。

## 决策驱动因素

- 防止 PID 复用、树结构竞态和权限变化造成误杀。
- 对工具已建立控制关系的进程提供可靠的正常停止和超时强制停止。
- 对外部进程采用保守、可解释且必须确认的操作。
- 重复或并发请求必须幂等。

## 决策

托管进程与外部进程使用不同的 RPC 方法、领域服务、确认 DTO 和 UI 文案。

### 托管进程

- 首先进入 `StopRequested`，重新校验 `ProcessInstanceKey` 后尝试正常停止。
- 正常停止期间进入 `GracefulStopping`，等待启动配置定义的超时。
- 超时后只能在用户请求或已明确允许的流程中进入 `ForceStopping`，再次校验身份，再使用创建时建立的控制边界强制停止托管树。
- 每个停止操作携带 `operationId`；重复请求返回同一操作状态，不重复发送信号或创建竞争状态机。
- 已退出返回 `ALREADY_EXITED` 或一致的已完成操作结果；身份变化返回 `IDENTITY_MISMATCH`，不得继续发送信号。

Windows 托管启动采用 `CREATE_SUSPENDED`，成功加入 Job Object 后才 `ResumeThread`。正常停止仅在条件满足的控制台任务中使用 `CREATE_NEW_PROCESS_GROUP` 与 `CTRL_BREAK_EVENT`；不适用或失败时进入超时流程。加入 Job Object 失败时不得继续运行失控进程。强制停止使用已建立的 Job 控制关系，不以递归 PID 枚举冒充进程树控制。

macOS 托管启动创建独立进程组。正常停止向工具创建且仍匹配的 PGID 发送 `SIGTERM`，超时后向同一受控组发送 `SIGKILL`。进程 daemonize 或主动脱离组时，标记为 `Orphaned` 或 `NotFullyManaged`，不得宣称已完整回收。

### 外部进程

- 默认只停止用户选中的单个 PID，不递归停止父子树。
- 每次操作都显示确认，并在实际发送信号前重新校验完整 `ProcessInstanceKey`。
- 禁止操作系统关键 PID、当前桌面应用和 Supervisor。
- 权限不足、不支持、目标已退出或身份不匹配时返回结构化错误，不尝试提权或降级强杀。
- V1 不承诺可靠停止外部进程的整棵进程树。

## 必须保持的不变量

- 托管停止和外部停止永远是不同接口，不得以布尔参数隐藏语义差异。
- 每次发送任何正常或强制停止信号前都必须重新校验完整实例身份。
- 外部停止默认且仅针对选中 PID，必须由用户确认。
- 托管树强制停止必须依赖创建时建立的 Job Object 或进程组控制关系。
- 不得以递归 PID 枚举替代平台控制边界。
- 不得停止系统关键 PID、当前应用或 Supervisor。
- 不得整体提权或绕过操作系统权限。

## 平台差异

### Windows

- 托管树控制基于 Job Object。
- `CTRL_BREAK_EVENT` 只用于满足控制台和进程组前置条件的托管任务。
- 外部进程操作基于先打开并校验的进程句柄。

### macOS

- 托管树控制基于工具创建的 PGID。
- 正常和强制停止分别使用 `SIGTERM` 和 `SIGKILL`。
- 脱离进程组的后代不能伪装为已受控或已回收。

## 被否决的方案

### 托管和外部进程共用一个 stop 接口

- 调用方无法明确表达控制证据和确认要求，容易误用强制树停止。

### 对外部进程默认递归停止后代

- 父子关系会变化，PID 会复用，扫描得到的树不能证明操作时的所有权。

### Windows 通过递归 PID 强杀实现托管树停止

- 存在不可消除的遍历竞态，且无法替代创建时的 Job Object 控制。

### 身份校验失败后按 PID 继续

- 可能停止无关的新进程，违反防误杀目标。

## 影响

### 正面影响

- 用户能够清楚区分可靠托管控制与保守外部操作。
- 停止状态机可审计、幂等，并能正确表达失败和部分控制。
- 将误杀风险限制在可验证的实例和控制边界内。

### 代价与限制

- UI 需要三种不同交互：托管正常停止、托管强制停止、外部停止。
- 某些托管任务可能脱离平台控制边界，只能标记为不完全受控。
- 某些外部进程即使用户可见也可能因权限或身份无法停止。

## 安全与隐私

停止属于高风险操作，必须在 Supervisor 中执行输入校验、身份复核、保护 PID 检查和操作幂等。UI 确认不能替代 Rust 边界验证。

## 兼容与迁移

运行记录需要保存控制类型、进程实例键、停止方式和恢复状态。旧记录若无法证明控制关系，只能进入 `Orphaned` 或其他非受控状态，不能自动升级为托管控制。

## 验证方式

- P0-T02、P0-T03 仅编译对应平台生命周期 Spike，并记录正常、强制、breakaway/脱组和失败路径。
- 生命周期实现执行对应平台 `cargo check`。
- 父子孙、重复停止、PID 复用和真实信号行为只在用户授权或 CI 中测试。
