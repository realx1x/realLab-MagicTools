# ADR-0003：安全且版本化的本地 IPC

- 状态：Accepted
- 日期：2026-07-14
- 决策者：Dev Process Manager 项目
- 关联任务：P0-T01、P0-T04、P2
- 替代：无
- 被替代：无

## 背景

Tauri Bridge 需要跨进程连接 Supervisor，并发送包含进程控制、配置和日志访问的敏感请求。本地 IPC 不是天然可信边界：同一机器上的其他用户或恶意本地进程不能获得控制能力；UI、Bridge 与 Supervisor 版本不一致时也不能继续按错误协议解释消息。

## 决策驱动因素

- 只允许当前登录用户连接当前用户的 Supervisor。
- 请求必须可鉴权、追踪、去重和超时。
- 消息拆包不能依赖 JSON 中可能出现的换行。
- UI 必须可靠检测事件丢失、乱序和重连。
- 不兼容版本必须明确失败，不能静默降级为不安全协议。

## 决策

### 传输与文件权限

- Windows 使用异步 Named Pipe，访问控制列表只允许当前用户 SID。
- macOS 使用 Unix Domain Socket；运行目录和 Socket 只允许当前用户访问，Socket 模式为 `0600`。
- Supervisor 每次启动生成新的高熵随机会话令牌。令牌文件和所在目录仅当前用户可读。
- 清理旧 Socket 时必须验证路径属于本应用运行目录，不得删除任意路径或非本应用文件。

### 鉴权与版本协商

- 建立连接后先执行协议版本协商和基于会话令牌的 challenge/response 握手。
- 在版本和鉴权成功前，不处理业务方法，也不发送敏感快照或日志。
- challenge 必须为每次连接随机生成，响应不得直接回传令牌，防止简单重放。
- 版本不兼容、鉴权失败和连接中断均返回或映射为结构化状态，并关闭该连接。

### 帧与信封

- 使用长度前缀帧承载 Serde JSON，禁止依赖换行符拆包。
- 对帧长度设置明确上限；超限、截断或非法 JSON 作为协议错误处理，不能无限分配内存。
- 请求信封至少包含 `protocolVersion`、`requestId`、`operationId`、`method` 和 `params`。
- 响应必须关联 `requestId`；有副作用的操作使用 `operationId` 去重。
- 协议支持请求超时、取消和结构化错误。

### 快照与事件顺序

- 快照和所有增量事件携带 Supervisor 内单调递增的 `revision`。
- UI 先订阅事件，再携带起始 revision 请求快照，以避免订阅和拉取之间出现空窗。
- revision 跳号、倒序或重连时，UI 必须丢弃局部派生状态并重新请求完整快照。
- revision 不用于进程身份校验。

### 错误码

协议至少支持：`INVALID_ARGUMENT`、`NOT_FOUND`、`ALREADY_EXITED`、`ACCESS_DENIED`、`IDENTITY_MISMATCH`、`NOT_SUPPORTED`、`SUPERVISOR_UNAVAILABLE`、`TIMEOUT`、`CONFLICT`、`STORAGE_ERROR`、`PLATFORM_ERROR` 和 `INTERNAL`。调用方不得依赖错误消息字符串判断分支。

## 必须保持的不变量

- 未完成版本协商和鉴权的连接不能调用任何业务方法。
- IPC 端点、运行目录和令牌文件不能对其他本地用户开放。
- 业务消息只能通过有上限的长度前缀帧传输。
- 请求标识、操作标识和 revision 的语义不能混用。
- Bridge 不能通过额外未鉴权通道绕过 Supervisor RPC。
- 协议错误必须失败关闭，不能回退到无鉴权或无版本模式。

## 平台差异

### Windows

Named Pipe 的安全描述符必须基于当前用户 SID 显式创建。不能依赖默认 ACL，也不能为方便调试授予 `Everyone` 或其他宽泛主体访问权。

### macOS

Unix Socket 必须位于当前用户专用目录，目录权限需先于 Socket 创建完成。Socket 创建后校验模式为 `0600`，旧端点清理限制在规范化后的应用目录内。

## 被否决的方案

### 无鉴权的 localhost TCP

- 端口可被其他本地进程连接或抢占，暴露面大于平台本地 IPC，且不符合当前用户访问边界。

### 仅依赖端点文件权限

- 文件权限是第一层防护，但不能替代连接级会话鉴权、重放防护和版本确认。

### 以换行分隔 JSON

- 字符串内容可包含换行，流式读取还会发生拆包和粘包，无法形成稳健边界。

### 版本不匹配时尽力解析

- 对进程控制协议进行猜测式兼容可能产生错误操作，必须明确拒绝不兼容版本。

## 影响

### 正面影响

- 本地其他用户无法直接获得 Supervisor 控制权。
- UI 可以可靠处理断线、重连、事件丢失和并发操作。
- 协议演进具备明确拒绝和兼容边界。

### 代价与限制

- 两个平台需要独立传输与权限适配。
- 需要管理令牌生命周期、帧上限、握手超时和版本兼容矩阵。
- UI 重同步时需要放弃局部状态并重新构建视图。

## 安全与隐私

会话令牌不得写入普通日志、诊断包、通知、SQLite 或 UI 状态。握手失败不得泄露令牌、路径、用户 SID 等敏感细节。诊断只记录脱敏后的错误类型和协议版本。

## 兼容与迁移

协议版本是连接建立前的强制协商项。兼容版本范围由 Supervisor 明确声明；没有共同版本时返回不兼容状态并停止业务通信。运行中升级需要与 Supervisor 所有权和更新交接方案联合处理。

## 验证方式

- P0-T04 实验代码执行 `cargo check`。
- P2 协议、Supervisor 和平台 IPC crate 执行各自 `cargo check --all-targets`。
- 其他本地用户连接失败、令牌权限和重放防护的实机测试，仅在用户授权或 CI 中执行。
