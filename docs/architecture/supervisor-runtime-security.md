# Supervisor Runtime Security

P2-T01 establishes a private runtime directory containing `supervisor.lock` and a raw 32-byte `session.token`. On Windows the directory is scoped to the current user and login session; on macOS it is scoped to the current user. The Supervisor acquires an exclusive file lock before replacing the token. A second instance in that scope maps lock contention to `AppError` with `ErrorCode::Conflict`; permission and I/O failures carry structured operation, path, and I/O-kind details.

The runtime root comes from the operating system's current-user record rather than mutable `LOCALAPPDATA` or `HOME` environment variables. On Windows, `ProcessIdToSessionId` supplies the stable session component; every runtime directory and sensitive file must be owned by the current process token's user SID and receives a protected DACL containing one full-control ACE for that SID. Directory and file handles reject reparse points before their DACL is applied, and lock handles do not share delete access. On macOS, path components are opened relative to validated owner-matching directory handles with `O_NOFOLLOW | O_DIRECTORY`. Application-owned directories and files have extended ACLs removed before their modes are set to `0700` and `0600` respectively.

P2-T03 uses the same SID and session to derive a stable Named Pipe endpoint. The server applies a non-inheritable protected SDDL DACL, rejects remote clients, verifies `GetNamedPipeClientSessionId`, replenishes the next listener before handing off a connection, and returns only a role-restricted authenticated connection. Pre-authentication frames are limited to 16 KiB and the complete handshake has a five-second timeout.

P2-T04 derives `supervisor.sock` from the macOS passwd record for the effective UID rather than `HOME`. The listener can only be constructed from a borrowed `SupervisorInstanceGuard`, so its authority to inspect or remove a stale endpoint cannot outlive the single-instance lock. It accepts only the exact socket name directly below the fixed runtime root. Before removing a stale path it uses no-follow metadata and rejects symlinks, non-sockets, and owner mismatches; after binding it applies mode `0600` and rechecks socket type, UID, mode, device, and inode. Drop removes the endpoint only while its socket type, UID, device, and inode still match the listener's bound object. Both client and server call `getpeereid` before the token handshake, active authenticated connections are limited to 16, and only the role-restricted authenticated connection escapes the five-second/16 KiB pre-authentication boundary.

The current macOS implementation is scoped to the effective UID and does not claim audit-session isolation between concurrent login sessions of the same account. Darwin exposes `LOCAL_PEERTOKEN`, but the pinned Rust `libc` does not expose the audit-token type or ASID conversion, while XNU declares `audit_session_self` as a private API returning a process-local Mach port name rather than a stable ASID suitable for a shared path. `LOCAL_PEERTOKEN` plus `audit_token_to_asid` linkage and behavior therefore require compilation and validation on the macOS Intel and Apple Silicon Runners before an audit-session boundary can be claimed.

The ACL, session check, and token proof establish current-user and current-session routing; they do not claim to isolate the Supervisor from malicious code already running as the same SID. Such code can generally read a current-user token file and is outside the V1 privilege boundary. The token is still useful for mutual proof, cross-restart freshness, and avoiding transmission of the secret over IPC.

The guard removes the token while still holding the lock, then unlocks. A stale token is never treated as ownership evidence. These tasks do not start the Supervisor, open a live pipe or socket, launch a process, or claim real cross-user/session validation; validation is compile-only.
